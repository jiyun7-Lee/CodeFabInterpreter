[04]  팀 구성 & 역할 분담
----------------------------------------------------------------

"팀은 총 네 명으로 구성되어 있습니다.

저는 1차에 Tokenizer를 맡았고,
2차에 추가 연산자 구현과 Negative TC 31개 추가를 담당했습니다.

김홍원님은 1차에 Expression Parser를,
2차에 Function과 Array 구현을 맡았습니다.

심민구님은 1차에 Statement Parser와 Checker를,
2차에 Static Binding과 Constant Folding 최적화를 담당했습니다.

이지윤는 1차에 Executor와 Shell을,
2차에 FileRunner, FactoryShell,
그리고 Breakpoint · Watch · Step을 포함한
Debug Mode 전체를 구현했습니다.

각자 1차에서 파이프라인 한 레이어씩 독립적으로 개발하고,
2차에서 기능을 확장하는 방식으로 진행했습니다."


----------------------------------------------------------------
[05]  전체 구조 UML
----------------------------------------------------------------

"이 UML은 CodeFabInterpreter의 전체 구조를 나타내며,
총 9가지 GoF 디자인 패턴이 적용되어 있습니다.
크게 네 개의 레이어로 나뉩니다.
왼쪽 위부터 시계 방향으로 설명드리겠습니다."

  ── 실행 모드 레이어 ───────────────────────────────────────

"먼저 파란색 박스, 실행 모드 레이어입니다.

진입점인 FactoryShell은 Factory Method 패턴입니다.
createRunner() 메서드 하나가 실행 인수를 분석해서,
REPL이면 ReplAdapter,
파일 실행이면 FileAdapter,
디버그 모드면 DebugAdapter를 생성합니다.
새로운 실행 모드가 생겨도 run() 로직을 건드릴 필요가 없습니다.

세 Adapter는 모두 IShellRunner 인터페이스를 구현합니다.
이게 Adapter 패턴입니다.
Shell::run()은 인수가 없고,
FileRunner::run(path)는 경로를 받는데,
Adapter가 이 서로 다른 시그니처를
IShellRunner::run() 하나로 통일시킵니다.

그 아래 BaseRunner가 Template Method 패턴입니다.
executeSource()가 파이프라인 순서,
즉 tokenize → parse → check → execute 순서를 고정해 놓고,
Shell, FileRunner, DebugShell은 입력 방식만 다르게 구현합니다."

  ── 파이프라인 레이어 ──────────────────────────────────────

"초록색 박스, 파이프라인 레이어입니다.

Tokenizer가 소스코드를 토큰으로 쪼개면,
Parser가 AST로 변환합니다.
Parser 안에 IExprParser 인터페이스가 보이는데,
이게 Strategy 패턴입니다.
표현식 파싱 알고리즘을 외부에서 주입받기 때문에,
테스트 시에는 FakeExprParser로 교체할 수 있습니다.

Checker는 정적 분석과 상수 폴딩을 담당하고,
변수가 몇 단계 스코프 위에 있는지 distance를 사전 계산해 놓습니다.

Executor에는 Observer Subject라고 표시되어 있습니다.
Stmt 하나를 실행하기 직전마다
debug_->beforeExecute()를 호출합니다.
Debug 모드일 때는 DebugController가 붙어 있어 이 이벤트를 받고,
일반 실행 모드일 때는 nullptr이라 오버헤드가 없습니다.
이게 Observer 패턴입니다."

  ── AST 레이어 ─────────────────────────────────────────────

"주황색 박스입니다. 이 레이어에 패턴이 세 개 겹쳐 있습니다.

먼저 Interpreter 패턴입니다.
Stmt와 Expr 계층 전체가 언어의 문법 트리를 직접 표현합니다.
AST 자체가 인터프리터 패턴의 구현체입니다.

Visitor 패턴은 ExprVisitor 인터페이스가 핵심입니다.
Executor가 이 인터페이스를 구현합니다.
표현식을 평가할 때 expr->accept(*this)를 호출하면,
구체 Expr 타입이 자신에게 맞는 visitXxx()를 역으로 호출합니다.
이중 디스패치로 dynamic_cast 체인을 완전히 제거했고,
새로운 Expr 타입을 추가했을 때
visitXxx 구현을 빠뜨리면 컴파일 오류가 납니다.

Composite 패턴은 BlockStmt, IfStmt, ForStmt에 적용됩니다.
이 세 클래스는 자식 Stmt를 내부에 보유합니다.
Executor는 Leaf인지 Composite인지 구분하지 않고 동일하게 처리합니다."

  ── Debug 레이어 ───────────────────────────────────────────

"분홍색 박스, Debug 레이어입니다.

DebugController가 Observer 패턴의 수신자입니다.
아까 말씀드린 것처럼 Executor로부터 이벤트를 받아
브레이크포인트 체크, 와치 변수 출력을 담당합니다.

ExecutionState 열거형이 State 패턴입니다.
네 가지 상태가 있습니다.
RUNNING은 자유 실행,
브레이크포인트에 도달하면 PAUSED로 전환됩니다.
STEP은 매 문장마다 정지,
NEXT는 현재 depth 이하의 문장만 통과합니다.
상태 전환 로직이 DebugController 안에 캡슐화되어 있어서
Executor는 상태를 몰라도 됩니다."

  ── Environment ────────────────────────────────────────────

"마지막으로 보라색 박스입니다.

Environment는 스코프 체인을 표현합니다.
각 Environment가 부모 포인터로 연결되는 이 구조가
Chain of Responsibility입니다.
단, 일반적인 Chain처럼 체인을 순회하면 O(n)이 됩니다.
여기서는 Checker가 사전에 distance를 계산해두기 때문에,
getAt(distance)로 정확한 스코프에 O(1)로 바로 접근합니다."

  ── 마무리 ─────────────────────────────────────────────────

"정리하면,
실행 모드 진입은 Factory Method와 Adapter,
파이프라인 순서 보장은 Template Method,
표현식 파싱 교체는 Strategy,
AST 평가는 Visitor와 Composite,
디버그 이벤트는 Observer,
디버그 상태 전환은 State,
스코프 탐색은 Chain of Responsibility로 설계되었습니다.
각 패턴이 독립적으로 적용된 것이 아니라
레이어별로 자연스럽게 연결되어 있다는 점이 특징입니다."

----------------------------------------------------------------
[07]  파이프라인 전체 흐름
----------------------------------------------------------------

"CodeFabInterpreter의 실행 흐름은 다섯 단계로 고정되어 있습니다.
BaseRunner.executeSource()가 이 순서를 보장합니다.

1행을 보시면,
소스코드가 들어오면 Tokenizer가 문자열을 토큰 열로 분리하고,
Parser가 토큰 열을 Abstract Syntax Tree로 변환합니다.

2행에서는 Checker가 정적 분석과 상수 폴딩을 수행하고,
Executor가 AST를 순회하며 실제 값을 계산해 출력합니다.
최종 출력은 '1'이 됩니다."


----------------------------------------------------------------
[08]  Parser — 토큰 열 → AST
----------------------------------------------------------------

"Parser는 재귀 하강 방식으로 토큰 열을 AST로 구성합니다.

상단의 다섯 토큰을 보시면,
PRINT, NUMBER:10, PERCENT, NUMBER:3, SEMICOLON 순서입니다.
Parser가 이를 읽으면서 PrintStmt 노드를 만들고,
그 아래 BinaryExpr에 % 연산자와 좌우 LiteralExpr를 붙입니다.

연산자 우선순위는 왼쪽 표에 정리되어 있습니다.
단항 연산자가 가장 높고, 대입이 가장 낮습니다.
재귀 하강 파싱은 이 우선순위를 함수 호출 깊이로
자연스럽게 표현합니다."


----------------------------------------------------------------
[09]  Checker — 정적 분석 + 최적화
----------------------------------------------------------------

"Checker는 실행 전에 세 가지 정적 분석을 수행합니다.
미선언 변수 참조, 자기 참조 초기화, 함수 인자 수 불일치를 잡아냅니다.
이 오류들은 실행 시점이 아니라 파싱 직후에 발견됩니다.

Static Binding은 Checker의 핵심 최적화입니다.
각 변수가 몇 단계 스코프 위에 있는지 distance를 사전에 계산해서
AST 노드에 기록해 둡니다.
Executor는 이 값을 이용해 이름 탐색 없이
getAt(0, 'x')처럼 직접 접근합니다.

Constant Folding은 컴파일 시점에 계산 가능한 식을
미리 결과 리터럴로 대체합니다.
'2 + 3'은 Lit(5)로 폴딩되고,
'0 + x'는 항등식으로 x로 단순화됩니다.
단, '9 % 0'처럼 0 나누기가 포함된 경우는
폴딩하지 않고 런타임에 에러를 발생시킵니다."


----------------------------------------------------------------
[10]  Executor — AST 실행
----------------------------------------------------------------

"Executor는 Checker를 통과한 AST를 순회하며 실제 값을 계산합니다.

스코프 관리는 Environment 체인으로 합니다.
전역 환경 위에 블록, for 루프, 함수 호출마다
새로운 Environment가 생성되고 parent 포인터로 연결됩니다.
Checker가 계산한 distance가 0 이상이면
getAt(d)로 직접 이동하고,
-1이면 체인을 순회하는 fallback을 씁니다.

주요 처리를 보시면,
산술·비교·논리 연산자 전체를 지원하고,
문자열 PLUS로 연결 연산을 지원합니다.
함수 호출은 fnEnv를 생성하고
ReturnSignal 예외로 반환값을 전달합니다.
배열은 Array(n) 빌트인으로 생성하고
인덱스 범위를 검사합니다."


----------------------------------------------------------------
[11]  Debug Mode — DebugController
----------------------------------------------------------------

"Debug Mode는 Executor의 실행 흐름에 훅을 걸어
줄별 변수 watch, breakpoint를 제공합니다.

ExecutionState가 네 가지 상태를 가집니다.
RUNNING은 자유 실행,
STEP은 줄마다 정지,
NEXT는 현재 블록 깊이 안에서만 이동,
PAUSED는 breakpoint에서 정지해 사용자 입력을 기다립니다.

BreakpointManager는 줄 번호를 등록·삭제하고,
실행 중 해당 줄에 도달하면 PAUSED로 전환합니다.
WatchManager는 감시 변수를 등록해두고
정지 시마다 값을 자동 출력합니다.

사용 가능한 명령어는
step, next, continue, break N, watch V 입니다.
시연에서 직접 확인하시겠습니다."

[15]  BP 1 — TDD
----------------------------------------------------------------

"첫 번째는 TDD입니다.
테스트를 먼저 작성하고, 구현이 따라오게 했습니다.
총 348개 TC를 작성했고, 레이어 간 통합 충돌은 0건이었습니다.

BP 사례 첫 번째는 추가 연산자 구현, Issue #27입니다.
이슈 등록 후 Tokenizer · Parser · Executor · Optimizer
네 레이어의 TC를 동시에 작성해 전부 Red로 만들었습니다.
그 다음 각 레이어를 순서대로 구현하면서 Green으로 통과시켰고,
마지막에 Executor의 SLASH · PERCENT 중복 0-검사 블록을
리팩터링으로 통합했습니다.

BP 사례 두 번째는 이슈 기반 TC 추가, Issues #28~#33, PR #37입니다.
일곱 개 모듈의 누락 TC를 이슈로 등록했습니다.
이 중 TC-FN-16,
함수 외부 return 오류 메시지에 줄번호가 포함되어야 한다는 케이스가
Red로 나오면서
Stmt.h, Parser.cpp, Checker.cpp 세 파일 수정을 유발했습니다.
이슈가 곧 실행 가능한 명세가 되고,
구현이 자동으로 따라오는 구조입니다."


----------------------------------------------------------------
[16]  BP 2 — PR 기반 코드 리뷰
----------------------------------------------------------------

"두 번째는 PR 기반 코드 리뷰입니다.
모든 변경은 PR로 제출하고, 최소 한 명의 리뷰를 거쳐
승인 후 master에 merge했습니다.

리뷰 사례 첫 번째입니다.
PR #39에서 epicpoem이
Checker.cpp에 '#include <cmath>'가 없다는 것을 발견했습니다.
MSVC에서는 우연히 빌드됐지만
GCC · Clang에서는 컴파일 오류가 납니다.
즉시 amend 후 force-push로 수정했습니다.

리뷰 사례 두 번째입니다.
PR #35에서 hahagil이
FileRunner가 줄마다 독립 Executor를 생성하는 구조를 발견했습니다.
앞 줄에서 선언한 변수가 다음 줄에서 소멸되는 Critical 버그입니다.
Executor와 Checker를 FileRunner의 멤버 변수로 승격해서
상태를 유지하도록 수정한 뒤 재승인을 받았습니다."


----------------------------------------------------------------
[17]  BP 3 — AI (Claude Code) 활용
----------------------------------------------------------------

"세 번째는 AI 활용입니다.
llvm-cov 커버리지 측정 자동화에 Claude Code를 활용했습니다.

문제 상황입니다.
기존에 쓰던 OpenCppCoverage는 line coverage만 제공해서
function · branch 단위의 취약 지점을 파악하기 어려웠습니다.
llvm-cov를 도입하려면 컴파일러 플래그와 링커 설정을 건드려야 해서
팀 전체 환경 세팅 부담이 컸습니다.
빌드 시스템을 건드리는 것은 리스크 대비 투자 비용이 큰 작업이었습니다.

컴파일러 플래그, 링커 설정, 실행 순서를 함께 구성하고 AI가 자동화하도록 markdown을 만들었습니다.
line · function · branch 세 지표를 한 번에 측정·리포트하는 시스템을 만들었습니다.
결과적으로 팀원 모두 one-click으로 실행할 수 있게 됐습니다.

----------------------------------------------------------------
[20]  회고
----------------------------------------------------------------

TDD로 각 레이어를 독립적으로 검증했기 때문에
통합 시 충돌이 한 건도 없었습니다.
코드 리뷰로 포터빌리티 버그를 조기에 발견했습니다.
역할 분리가 명확해서 병렬 개발이 가능했고
